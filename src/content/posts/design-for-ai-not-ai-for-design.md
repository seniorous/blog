# Design for AI, Not AI for Design

副标题：看完 OpenAI《Harness Engineering》后，我对工程角色变化的一些判断

我读完那篇文章后，最先记住的不是“100 万行代码”或者“1500 个 PR”。这些数字当然醒目，但它们更像结果，不像原因。真正让我在意的是另一个变化：当写代码这件事越来越便宜，工程里最贵的部分就换了地方。贵的开始变成上下文、验证、约束、文档，还有团队能不能把一套规则长期维持住。

这也是我想重新写这篇笔记的原因。很多人会把 agent 看成更强的代码补全，或者更会干活的外包实习生。我现在越来越觉得，这种理解太窄了。agent 真正改变的不是“谁来写函数”，而是工程系统到底该围着什么来设计。

**TL;DR**：这篇博客最重要的结论，不是 AI 可以替程序员写多少代码，而是工程团队必须先把环境设计好，再谈把执行权交给 agent。代码会越来越便宜，坏环境会越来越贵。以后区分一个团队强不强，看的不只是代码风格，还要看它有没有能力把知识、规则、反馈和清理机制都落到仓库里。

![](/images/design-for-ai-not-ai-for-design/harness-01-shift.svg)

*图 1：变化的重点不只是“谁在写代码”，而是工程关注点从 code-first 转向环境与约束。*

### Agent环境

OpenAI 那篇文章里最容易被转发的是几个数字：从空仓库起步，五个月左右，做出一个内部 beta，代码量接近百万行，而且他们刻意坚持“人工不直接写代码”。我倒觉得，真正值得记住的是那个约束本身。只要你把“人不下场写代码”当成硬条件，很多问题就会立刻暴露出来。

比如，agent 为什么老在同一个地方犯错？很多时候不是模型太笨，而是环境太差。它看不到页面真实长什么样，看不到日志链路，看不到哪条规则只是历史包袱，也不知道仓库里哪些目录能碰、哪些目录只是暂存区。人在这种环境里尚且会踩坑，agent 只会把坑踩得更快。

所以我现在会把“agent 环境”理解得更具体一些。不是给它配个 IDE，接几个工具就结束了，而是至少要解决三件事：它能不能看到结果，它能不能读懂系统，它能不能在规则边界里行动。少任何一个，产出都会开始漂。

这也是我读完后的第一个判断：以后工程师很大一部分工作，不是亲手搬砖，而是在造施工现场。你要先决定路怎么走、材料放哪、验收标准是什么、出了事谁能追溯。agent 只是那个一直在干活的人，但现场秩序还是人定的。

### Agent如何闭环？

一旦生成速度不再稀缺，验证就会变成新的瓶颈。这件事看起来很朴素，真正做起来却很难。因为过去很多团队默认的工作方式是“先把东西写出来，再让人慢慢 review”。agent 进来以后，这套节奏会很快失灵。它不是写得慢，它是写得太快；不是没东西可改，而是改动多到你根本不想逐行盯。

所以闭环能力比生成能力更重要。一个能自己跑测试、看截图、查日志、对照指标的 agent，和一个只会提交 patch 的 agent，根本不是同一类工具。前者是在一个受约束的回路里工作，后者只是把更多不确定性推给人。

我很认同那篇文章里隐含的目标：理想状态下，agent 至少应该能自己值一段夜班。不是神话式的“全自动开发”，而是它碰到问题时，能先复现，能找证据，能做一次像样的修复，再把过程交还给人复核。这个顺序很重要。不是人替它闭环，而是它先闭环，再由人决定是否接收。

如果把这件事拆开看，浏览器、截图、日志、trace、指标、CI、worktree 其实都不是附属品，它们共同组成了 agent 的感觉器官。少一项都能凑合，多半也能跑；但只要任务复杂一点，它迟早会退回到“瞎猜”。

![](/images/design-for-ai-not-ai-for-design/harness-02-feedback-loop.svg)

*图 2：当生成足够便宜，反馈回路本身就成了生产力。*

### agent如何发现知识

很多团队接入 agent 的第一反应，是写一个巨大的 `AGENTS.md`，恨不得把所有约定一次性塞进去。这个做法我现在基本不信了。不是因为文档不重要，恰恰相反，是因为文档太重要了，不能用这种堆砌的方式处理。

大而全的总纲通常有两个结局。第一种是没人看完。第二种是大家都以为别人会维护它，最后它悄悄过期。对 agent 来说，过期文档比没有文档更糟，因为它真的会把那些过时规则当真，然后持续复制。

更稳的做法，是把仓库本身做成一张能导航的地图。`AGENTS.md` 只负责索引和行动约束，告诉 agent 应该先看哪里；`ARCHITECTURE.md` 负责交代系统分层和边界；真正细的规则、ADR、运行手册、接口约定，放进 `docs/`，让结构自己说话。

这背后其实是个很老派的工程判断：知识要能被定位，能被校验，也要能被淘汰。以前文档烂一点，主要是新同学痛苦；现在文档烂一点，等于你把错误输入接到了一个高吞吐复制器前面。坏知识会扩散，而且速度很快。

另外我越来越觉得，成熟和无聊的技术在 agent 时代会重新变得性感。API 稳定、约定清楚、生态大、历史案例多，这些东西以前只是“保守”；现在它们意味着更强的可读性和更低的歧义。对人友好的系统，往往也更容易被 agent 正确理解。

![](/images/design-for-ai-not-ai-for-design/harness-03-knowledge-map.svg)

*图 3：知识不该是一篇超长提示词，而应该是一套能导航、能校验、能维护的仓库地图。*

### 工程品味如何传达给agent

我以前总觉得“工程品味”这件事很难落地。它像一种经验判断，存在于 code review、口头提醒和少数老同学的直觉里。agent 出现以后，这个问题一下子变得刺眼了。因为 agent 不会自动领会“这个写法虽然能跑，但我们平时不这么写”。

换句话说，靠氛围是不够的。建议如果不能落成规则，很快就会失效。你可以跟人类工程师说“这个模块尽量别跨层调用”，人类会结合语境做判断；agent 只会看仓库里什么东西已经存在，然后照着扩写。它不怕坏模式，它只怕没有模式。

所以“工程品味如何传达”这个问题，最后大多会变成“哪些东西必须从建议升级成法律”。依赖方向要不要锁死，目录边界要不要测，接口格式要不要统一，文件大小要不要限制，哪些生成代码必须附测试，哪些改动一律不能直接落主干。真正有效的答案，通常都写在 lint、结构测试、脚本和 CI 里。

这一点我很赞同 OpenAI 的判断：与其不停修已经长歪的代码，不如优先修会稳定长歪的条件。你今天多写一条可执行规则，往后可能就少做几十次重复 review。对人来说，这叫约束；对 agent 来说，这更像扶手。

![](/images/design-for-ai-not-ai-for-design/harness-04-rules-as-law.svg)

*图 4：把品味写成机器能执行的边界，比反复人工纠偏更靠谱。*

### AGENT工作流程

传统流程默认“写代码”很贵，所以大家愿意把很多判断前置到 review 阶段，宁可慢一点，也要尽量一次过。agent 时代这笔账开始反过来算了。生成和修补都便宜了，等待反而变贵。你让一堆小问题在 review 阶段排队，本质上是在浪费最贵的东西: 时间和上下文切换。

这不意味着可以放弃审查，而是工作流要改。更小的 PR、更短的链路、更快的返工，会比那种憋一个大改动再全面会审的方式更适合 agent。因为 agent 最擅长的，不是一次把所有东西想明白，而是快速迭代、快速试错、快速回补。

但这里有个前提，不能省。只有在护栏足够密的时候，快才是优势；否则快只是把混乱更早推上主干。我觉得很多团队看那篇文章时，只盯住了“吞吐提升”，却没注意它背后的约束密度。真正难学的从来不是速度，而是让速度不把系统撞散。

所以如果要我自己总结一条工作流原则，那就是：不要把所有风险都堆在合并前，也不要把所有责任都堆给 reviewer。让 agent 在提交前尽可能自证，让系统在合并后持续兜底，让人只处理真正需要判断的地方。

![](/images/design-for-ai-not-ai-for-design/harness-05-cost-inversion.svg)

*图 5：在 agent 工作流里，最贵的成本不再只是“写”，而是“等”。*

### ai自主性可以到哪一步

如果只把 agent 当成“更会补全代码”的工具，那这篇博客其实没什么新鲜的。它真正讨论的是更长的一条链: 写代码、配 CI、补文档、回 PR、写脚本、改内部工具，甚至做一部分运维和清理工作。不是一个聪明编辑器，而是一种端到端的执行能力。

这也会把工程师的角色往前推。以后很多时候，工程师更像是在定任务、搭环境、写验收条件、决定边界，而不是亲自把每个细节敲出来。说得再直白一点，人越来越像系统设计者和审稿人，agent 越来越像那个不知疲倦的执行层。

我对“自主”这个词一直有点保留，因为它很容易让人误会成“放手不管”。现实恰好相反。自主性越高，对环境设计的要求越严。没有状态检查、没有回放能力、没有证据链、没有明确退出条件，所谓自主往往只是一种更昂贵的碰运气。

所以我能接受的自主，不是“你去把这个产品做出来”，而是“在明确目标、明确边界、明确验证方式的前提下，你自己把这一段任务跑完”。范围一模糊，自主就会退化成混乱。

### 如何控制agent的代码熵？

人类工程师会留下技术债，agent 会把技术债制度化。它最擅长延续现有模式，不管那个模式是优雅还是将就。仓库里只要有一处脏办法活得很好，它很可能很快就变成十处、二十处。

这也是为什么我觉得“代码熵”在 agent 时代不是抽象问题，而是运营问题。你不能只靠年末大扫除，也不能等系统明显变坏了再回头修。更实际的办法，是尽量提供稳定的公共能力，让 agent 少造轮子；把高风险边界验证前置，别让它一路猜过去；再加上一些固定频率的清理动作，把 slop 当成真的维护对象。

OpenAI 提到让后台 agent 定期扫仓库、找坏模式、开修复 PR。我很认同这个方向。不是因为它听起来高级，而是因为它符合现实。既然 agent 会持续制造局部混乱，那清理就不能靠偶然心情，得像跑测试、发版本一样常态化。

判断一个团队成熟不成熟，以后恐怕也不只是看能不能把功能做出来，而要看它有没有一套机制，能压住坏模式复制的速度。代码风格只是表面，复制速度才是底层变量。

*图 6：agent 会放大仓库里已经存在的一切，好的坏的都一样。*

### 对于个人开发者

如果你没有 OpenAI 那样的基础设施，这篇文章也不是只能拿来感叹。个人开发者其实更适合先把这套思路跑起来，因为决策链短，很多事今天改、明天就能看到效果。

我会建议从下面几个地方下手。第一，先把最小闭环补齐。哪怕只是本地预览、自动测试、基础日志和错误回放，也比让 agent 纯靠猜强得多。第二，把仓库写得能读。一个短一点的 `AGENTS.md` 索引，一个说明模块边界的 `ARCHITECTURE.md`，再把关键约定放进 `docs/`，已经能解决很多低级误判。第三，不要一开始就追求“全自动开发”。先抓一条最重要的不变量，比如依赖方向、接口格式或者目录边界，把它写成脚本、lint 或测试。第四，把任务切小。一次只让 agent 动一个局部，远比让它同时碰结构、样式、数据和部署稳得多。第五，记得维护文档。文档现在不是交付后的附属品，而是运行时环境的一部分。

我现在越来越倾向于把整件事概括成一句很土的话：别急着把 AI 塞进你的设计流程里，先把你的工程系统整理成适合 AI 工作的样子。Design for AI, not AI for design。顺序反了，最后通常不是提效，而是更快地把混乱放大。
